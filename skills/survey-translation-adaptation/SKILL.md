# Survey Translation & Adaptation

Produce language versions that measure the same construct the same way across groups. Literal translation is the default failure mode; the goal is equivalence, which often requires adaptation.

## Load first
- `../../shared/3mc-considerations.md` — the comparability lens.
- `../../shared/tse-framework.md` — translation errors live in the measurement half and feed comparison error.

## Translation vs. adaptation
- **Translation:** render meaning in another language.
- **Adaptation:** change content/format when a literal version would not be equivalent (idioms, culturally specific concepts, response scales that don't map, examples that don't exist locally).
Decide per item which you're doing, and document why.

## Process — TRAPD
1. **Translation** — two or more independent translators per language (not one).
2. **Review** — translators + a survey methodologist reconcile versions item by item.
3. **Adjudication** — a final decision-maker signs off each item; resolves disputes.
4. **Pretesting** — cognitive interviewing in each language to catch inequivalence the team missed.
5. **Documentation** — record every decision; comparability rests on this trail.

Back-translation is a *check*, not the method — useful to surface problems, not a substitute for TRAPD.

## Harmonization
Decide input vs. output harmonization (see TSE file). For shared languages across regions, harmonize so the same language doesn't fork into incompatible versions.

## Output
Per-language instrument versions linked to source item IDs in the instrument library, each item tagged translation vs. adaptation, plus a decisions log and a list of items still needing pretesting.
